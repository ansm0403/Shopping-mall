## 1. 사전 질문 — TypeORM vs ElasticSearch

**사용자:**
> ElasticSearch를 이용한 검색 기능을 구현하려고 한다. 우선 작업에 들어가기 전에 궁금한 점부터 질문하겠다.
> TypeORM의 where 등의 구문만으로도 검색은 구현할 수 있다고 들었다. 물론 ElasticSearch의 강력한 구문 분석기 등의 기능은 사용할 수 없겠지만.
> 만약 TypeORM만으로 검색기능을 구현한 후, 그 다음 필요할 때 ElasticSearch 구현이 가능하도록 확장성 있게 검색기능을 제작할 수 있는가?
> 그리고 나는 검색에 제품명뿐 아니라 태그까지 사용하고 싶은데 TypeORM만으로는 불가능한가?

**Claude:**

README와 entity 파일들을 확인한 후 다음과 같이 답변했다.
(`Product ── M:M ── Tag` 구조 확인)

### 1. TypeORM만으로 검색 가능한가?

**가능하다.** TypeORM의 `ILike`(대소문자 무시 LIKE)로 부분 일치 검색을 구현할 수 있다.

```typescript
where: { name: ILike(`%${keyword}%`) }
```

다만 TypeORM의 한계:
- 형태소 분석 없음 → "무선 이어폰" 검색 시 "이어폰 무선"은 미매칭
- 오타 허용(fuzzy) 없음
- 관련성 점수(relevance scoring) 없음

### 2. 태그 검색 — TypeORM으로 가능한가?

**가능하다. 단, QueryBuilder가 필요하다.** `Product ── M:M ── Tag` 구조에서 단순 `where` 객체 방식은 JOIN을 못 쓴다.

```typescript
queryBuilder
  .leftJoin('product.tags', 'tag')
  .where('product.name ILIKE :keyword', { keyword: `%${q}%` })
  .orWhere('tag.name ILIKE :keyword', { keyword: `%${q}%` });
```

### 3. ElasticSearch로 나중에 확장 가능한가?

**충분히 가능하다.** 핵심은 **SearchService 인터페이스를 추상화**하는 것이다.

```typescript
interface ISearchService {
  searchProducts(query: ProductSearchQuery): Promise<SearchResult>;
}

// 현재: TypeORM 구현체
class TypeOrmSearchService implements ISearchService { ... }

// 나중에: ElasticSearch 구현체로 교체
class ElasticSearchService implements ISearchService { ... }
```

ProductService는 `ISearchService`에만 의존하도록 작성하면, 나중에 ES 구현체로 교체할 때 ProductService 코드를 건드릴 필요가 없다.

### 정리

| 기능 | TypeORM | ElasticSearch |
|------|---------|--------------|
| 부분 일치 검색 | `ILike` ✅ | ✅ |
| 태그 검색 | QueryBuilder JOIN ✅ | ✅ |
| 한국어 형태소 분석 | ❌ | nori 플러그인 ✅ |
| 오타 허용(fuzzy) | ❌ | ✅ |
| 관련성 점수 | ❌ | ✅ |

## 3. 옵션 A vs B 결정

**사용자:**
> 니가 생각하기에는 옵션 B와 A 중 어떤 것이 더 좋을 것이라고 생각하나? 효율과 성능, 복잡성 등을 기준으로 알려줘. 아직 초보자인 나는 판단이 안 서네.

**Claude:**

**옵션 B를 추천한다.** 이유:

| 기준 | 옵션 A | 옵션 B |
|------|--------|--------|
| 코드 중복 | ❌ 없음 | ⚠️ 약간 (페이지네이션) |
| CommonService 복잡도 | ❌ 증가 | ✅ 그대로 |
| 버그 위험 | ⚠️ 높음 (기존 기능 영향) | ✅ 낮음 |
| ElasticSearch 확장 | ⚠️ 불편 | ✅ 깔끔 |
| 초보자 이해도 | ⚠️ 어려움 | ✅ 쉬움 |

**핵심 이유:**

1. **기존 코드 영향 최소화** — CommonService는 지금 잘 작동 중. 손대면 버그 위험 증가
2. **ElasticSearch 교체 시 깔끔** — `ProductSearchService` 통째로 교체, Module에서 `useClass`만 변경
3. **포트폴리오 관점** — 인터페이스 기반 설계는 시니어 개발자들이 보기에 좋은 신호
4. **코드 중복은 무시해도 됨** — DRY < 명확성(Clarity). 나중에 필요하면 공통화 가능

구현 방향:
```typescript
// 지금
{ provide: PRODUCT_SEARCH_SERVICE, useClass: TypeOrmProductSearchService }

// 나중에
{ provide: PRODUCT_SEARCH_SERVICE, useClass: ElasticProductSearchService }
```

### 아키텍처 흐름

```
GET /v1/products?keyword=무선이어폰&tags=여름,세일&page=1&take=20
```

### Task 2 — ProductQueryDto 수정

`keyword`와 `tags` 필드 추가:

```typescript
/** 제품명 + 태그명 통합 검색어 */
@IsOptional()
@IsString()
@Transform(({ value }) => (typeof value === 'string' ? value.trim() : value))
keyword?: string;

/**
 * 특정 태그 필터 (AND 조건)
 * 쿼리스트링: tags=블루투스,여름 → ['블루투스', '여름']
 */
@IsOptional()
@IsArray()
@IsString({ each: true })
@Transform(({ value }) =>
  typeof value === 'string'
    ? value.split(',').map((v: string) => v.trim()).filter(Boolean)
    : value,
)
tags?: string[];
```

### Task 3 — TypeOrmProductSearchService 구현

`backend/src/product/product-search.service.ts` 신규 생성.

핵심 로직:

```typescript
// 키워드: 제품명 OR 태그명 ILIKE
qb.andWhere(
  new Brackets((sub) => {
    sub
      .where('product.name ILIKE :keyword', { keyword: `%${query.keyword}%` })
      .orWhere('tag.name ILIKE :keyword', { keyword: `%${query.keyword}%` });
  }),
);

// 특정 태그 필터 (AND 조건) — EXISTS 서브쿼리 방식
query.tags.forEach((tagName, idx) => {
  qb.andWhere(
    `EXISTS (
      SELECT 1 FROM product_tags pt${idx}
      JOIN tags filteredTag${idx} ON filteredTag${idx}.id = pt${idx}.tag_id
      WHERE pt${idx}.product_id = product.id
        AND filteredTag${idx}.name = :tagName${idx}
    )`,
    { [`tagName${idx}`]: tagName },
  );
});
```

## 6. 엣지 케이스 보강

**사용자:**
> 이제 이 로직에서 발생할 수 있는 엣지 케이스(예: DB 트랜잭션 실패, 잘못된 입력값 등 중요한 로직) 3가지만 뽑아서 예외 처리 코드를 보강해줘. 3가지가 부족하다 싶으면 추가해도 좋다.

**Claude:**

코드를 분석한 결과 **4가지** 엣지 케이스를 발견했다.

| # | 문제 | 위험도 |
|---|------|--------|
| 1 | **sortBy 검증 없음** — 허용되지 않은 컬럼 이름으로 SQL 에러 발생 가능 | 중간 |
| 2 | **빈 키워드** — `" "` → trim 후 `""` → `ILIKE '%%'` → 전체 상품 반환 | 중간 |
| 3 | **M:M JOIN 중복** — 태그 2개 매칭 시 같은 상품이 2번 나옴 (count는 방어했지만 data는 미방어) | 높음 |
| 4 | **잘못된 커서** — `applyCursor`에서 base64 디코딩/JSON.parse 실패 시 500 에러 | 높음 |

### EC1. sortBy 검증 (신규 `validateSortBy()` 메서드)

```typescript
private validateSortBy(sortBy: string): SortableKey {
  if (!SORTABLE_COLUMNS.includes(sortBy as SortableKey)) {
    throw new BadRequestException(
      `sortBy는 ${SORTABLE_COLUMNS.join(', ')} 중 하나여야 합니다.`,
    );
  }
  return sortBy as SortableKey;
}
```

`GET /v1/products?keyword=이어폰&sortBy=password` 같은 요청에서 SQL 에러 대신 명확한 400 반환.

### EC2. 빈 키워드 방어 (`search()` 진입부)

```typescript
if (!query.keyword || query.keyword.trim().length === 0) {
  throw new BadRequestException('검색어(keyword)를 입력해주세요.');
}
```

`GET /v1/products?keyword=   ` → `ILIKE '%%'`로 전체 상품 반환되는 것을 방지.

### EC3. M:M JOIN 중복 방어 (2단계 쿼리 전략)

**기존 방식의 문제:**
```
product "무선이어폰" + tag "블루투스" + tag "여름"
→ LEFT JOIN 결과: product row 2개
→ getMany() 중복 반환
```

**수정: 2단계 쿼리 전략**
```typescript
// 1단계: DISTINCT ID만 추출
const idQb = qb.clone()
  .select('DISTINCT product.id', 'id')
  .orderBy(...)
  .limit(take);
const idRows = await idQb.getRawMany();
const ids = idRows.map((r) => r.id);

// 2단계: relation 포함 로드 (중복 없음)
const data = await this.productRepository
  .createQueryBuilder('product')
  .leftJoinAndSelect('product.tags', 'tag')
  .leftJoinAndSelect(...)
  .whereInIds(ids)
  .getMany();
```

### EC4. 커서 디코딩 실패 방어

```typescript
private applyCursor(...): void {
  let decoded: { sortValue: unknown; id: unknown };
  try {
    decoded = JSON.parse(Buffer.from(cursor, 'base64').toString('utf-8'));
  } catch {
    throw new BadRequestException('유효하지 않은 커서 형식입니다.');
  }

  if (decoded.sortValue === undefined || decoded.id === undefined) {
    throw new BadRequestException('커서에 필수 값(sortValue, id)이 없습니다.');
  }
  // ...
}
```

`GET /v1/products?keyword=이어폰&cursor=잘못된base64값` → 500 에러 대신 400 반환.

| `product/dto/product-response.dto.ts` | 수정 | `tags` 필드 노출 |

### 사용 예시

```
# 제품명 + 태그명에서 검색
GET /v1/products?keyword=무선이어폰

# 검색 + 특정 태그 필터 (AND)
GET /v1/products?keyword=이어폰&tags=블루투스,여름

# 검색 + 가격 필터 + 페이지
GET /v1/products?keyword=이어폰&filter[price][lte]=50000&page=1

# 검색 + 커서 기반 무한스크롤
GET /v1/products?keyword=이어폰&sortBy=price&sortOrder=ASC
```

### 핵심 설계 원칙

1. **기존 API 호환성 유지** — keyword 없는 기존 요청은 CommonService.paginate() 그대로
2. **인터페이스 기반 확장성** — `PRODUCT_SEARCH_SERVICE` 토큰의 `useClass`만 교체하면 ElasticSearch로 전환 가능
3. **M:M JOIN 중복 방어** — 2단계 쿼리 전략 (DISTINCT ID 추출 → whereInIds 로드)
4. **명확한 에러 처리** — 4가지 엣지 케이스 모두 500 → 400으로 변환
