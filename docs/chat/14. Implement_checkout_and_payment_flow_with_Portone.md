# 14. 구매/결제 기능 구현 (포트원 V2 연동)

> 장바구니 기능 완성 이후, 주문서 작성 → 결제 → 주문 조회 → 취소/환불 → 구매 확정까지 전체 구매 플로우를 구현한 기록입니다. 처음에는 포트원 V1으로 구현한 뒤, V2로 마이그레이션하고 엣지케이스를 2회에 걸쳐 수정했습니다.

---

## 1. 설계

### 사용자 요청

> 장바구니 기능을 만들었으니 이제 구매 기능을 구현하고 싶다. 아래 7가지 기능을 설계해달라.

### 구현 대상 기능

| # | 기능 | 설명 |
|---|------|------|
| 1 | 주문서 작성 페이지 | 장바구니 데이터 표시, 배송지/수령인 입력, 결제 요약 |
| 2 | 결제 (포트원 연동) | 결제창 호출 → 결제 완료 → 서버 검증 |
| 3 | 주문 목록 | 상태별 필터 탭, 페이지네이션 |
| 4 | 주문 상세 | 진행 단계 표시, 상품/배송/결제 정보, 액션 버튼 |
| 5 | 주문 취소 | PENDING_PAYMENT 상태의 미결제 주문 취소 |
| 6 | 결제 취소/환불 | PAID/PREPARING 상태의 결제 완료 주문 환불 |
| 7 | 구매 확정 | DELIVERED 상태에서 사용자가 직접 확정 |

### 사용 백엔드 API

| 메서드 | 경로 | 설명 |
|--------|------|------|
| POST | /orders | 주문 생성 (장바구니 → 주문 변환) |
| GET | /orders/my | 내 주문 목록 (상태 필터, 페이지네이션) |
| GET | /orders/my/:orderNumber | 주문 상세 조회 |
| PATCH | /orders/:orderNumber/cancel | 주문 취소 (미결제) |
| PATCH | /orders/:orderNumber/confirm | 구매 확정 |
| POST | /payments/verify | 결제 검증 (포트원 ↔ DB 금액 비교) |
| POST | /payments/:orderNumber/cancel | 결제 취소/환불 |

### 설계 포인트

- **금액 위변조 방지**: 프론트엔드가 계산한 금액이 아니라 백엔드가 저장한 `totalAmount`를 결제창에 전달
- **이중 결제 방지**: `isProcessingRef`로 전체 비동기 흐름(주문 생성 → 결제 → 검증) 잠금
- **멱등성**: 서버에서 `SELECT FOR UPDATE`로 동시 요청 처리, 이미 PAID면 기존 결과 반환
- **이벤트 드리븐**: 결제 완료 → `order.paid` 이벤트 → salesCount 증가 + Shipment 생성

### 핵심 코드 설명

#### 결제 흐름 (checkout/page.tsx)

```
사용자가 "결제하기" 클릭
  → isProcessingRef 잠금 (이중 클릭 방지)
  → 1단계: createOrder API 호출 (백엔드에서 주문 + Payment 레코드 생성)
  → 2단계: PortOne.requestPayment() 호출 (결제창 표시)
  → 3단계: verifyPayment.mutate() 호출 (서버에서 포트원 ↔ DB 금액 검증)
  → 성공: 주문 상세 페이지로 이동
  → 실패: alert + 주문 상세 페이지로 이동 (PG 결제는 성공했을 수 있으므로)
```

#### 서버 결제 검증 (payment.service.ts - verifyPayment)

```
1. Payment 레코드 조회 (paymentId = orderNumber)
2. 본인 주문 확인
3. 멱등성 가드: 이미 PAID면 기존 결과 반환
4. 만료/취소 주문에 결제가 된 경우 → 자동 환불
5. 포트원 V2 API로 실제 결제 정보 조회
6. 금액 비교 (amount.total vs DB 금액) → 불일치 시 자동 취소
7. 트랜잭션: SELECT FOR UPDATE → Payment/Order 상태 갱신
8. order.paid 이벤트 발행 → salesCount 증가 + Shipment 생성
```

## 3. 포트원 V1 → V2 마이그레이션

### 사용자 요청

> 포트원 식별코드가 V1과 V2 두 개가 있다. 현재 코드가 어떤 버전인지, V2로 바꾸면 변경량이 많은지 알려달라.

### V1과 V2의 차이

| 항목 | V1 | V2 |
|------|----|----|
| 인증 방식 | imp_key + imp_secret → access_token (2단계) | API Secret 직접 사용 (1단계) |
| API 서버 | api.iamport.kr | api.portone.io |
| 프론트엔드 SDK | `<script>` 태그 + `IMP.request_pay()` 콜백 | `@portone/browser-sdk` npm + `PortOne.requestPayment()` async/await |
| 결제 상태값 | 소문자 (`paid`, `cancelled`) | 대문자 (`PAID`, `CANCELLED`) |
| 금액 필드 | `amount` (숫자) | `amount.total` (객체) |
| 날짜 필드 | Unix 타임스탬프 (`paid_at * 1000`) | ISO 문자열 (`new Date(paidAt)`) |
| 웹훅 이벤트 | `status === 'paid'` | `type === 'Transaction.Paid'` |

### 변경 범위

**백엔드 (3파일):**
- `payment.service.ts`: 전면 재작성 — 토큰 교환 로직 제거, V2 API 호출 구현, 응답 매핑 변경
- `webhook-payment.dto.ts`: V2 웹훅 구조 (`{ type, data: { paymentId, transactionId? } }`)
- `.env`: `IMP_KEY`/`IMP_SECRET` → `PORTONE_API_SECRET`

**프론트엔드 (2파일):**
- `checkout/page.tsx`: `iamport.js` Script → `@portone/browser-sdk/v2` import, `IMP.request_pay()` 콜백 → `PortOne.requestPayment()` async/await
- `model/order.ts`: 타입 업데이트

### DB 스키마 변경 없이 마이그레이션

TypeScript 프로퍼티명만 변경하고 DB 컬럼명은 유지:

```typescript
// payment.entity.ts
@Column({ name: 'imp_uid', nullable: true })
transactionId: string | null;  // 기존: impUid

@Column({ name: 'merchant_uid', unique: true })
paymentId: string;  // 기존: merchantUid
```

이렇게 하면 기존 DB 데이터와의 호환성을 유지하면서 코드에서는 V2 용어를 사용할 수 있습니다.

### 연쇄 변경이 필요했던 파일들

entity 프로퍼티명이 바뀌면서 이를 참조하는 파일들도 함께 수정해야 했습니다:

| 파일 | 변경 내용 |
|------|-----------|
| `verify-payment.dto.ts` | `impUid` → `transactionId`, `merchantUid` → `paymentId` |
| `payment-response.dto.ts` | `impUid` → `transactionId`, `merchantUid` → `paymentId` |
| `order.service.ts` | `merchantUid: orderNumber` → `paymentId: orderNumber` |
| `order-response.dto.ts` | PaymentSummaryDto 필드명 변경 |
| `order.events.ts` | `OrderPaidEvent.impUid` → `transactionId` |
| `order-event.listener.ts` | 로그 메시지의 `impUid` → `transactionId` |

**교훈**: 프로퍼티명을 변경할 때는 `grep`으로 프로젝트 전체를 검색해서 참조하는 곳을 모두 찾아야 합니다. IDE의 "Find All References"도 좋지만, DTO나 이벤트 클래스처럼 직접 import하지 않는 곳은 놓칠 수 있습니다.

## 4. 엣지케이스 분석 및 수정

### 1차 엣지케이스 (6건)

#### 1. 이중 결제 방지 (checkout/page.tsx)

**문제**: `createOrderMutation.isPending`으로 버튼을 비활성화했지만, 주문 생성 성공 후 `isPending`이 `false`가 되면서 결제창이 열리기 전에 다시 클릭 가능.

**원인**: React Query의 `isPending`은 해당 mutation의 생명주기만 추적합니다. 하지만 우리 결제 흐름은 "주문 생성 → 결제창 → 결제 검증"이라는 3단계로, 한 mutation이 끝나도 전체 프로세스는 진행 중입니다.

**수정**: `useRef(false)`로 전체 비동기 흐름을 잠금.

```typescript
const isProcessingRef = useRef(false);

const handleSubmit = async () => {
  if (isProcessingRef.current) return;  // 이미 진행 중이면 무시
  isProcessingRef.current = true;

  try {
    // 1단계: 주문 생성
    // 2단계: 결제창
    // 3단계: 검증
  } catch {
    isProcessingRef.current = false;  // 실패 시만 리셋
  }
};
```

**왜 `useState` 대신 `useRef`?**: `useState`는 상태 변경 시 리렌더링을 일으킵니다. 이미 `isSubmitting` 상태가 버튼 비활성화를 담당하고 있으므로, 순수한 잠금 용도로는 리렌더링이 없는 `useRef`가 적합합니다.

#### 2. verifyPayment 실패 시 사용자 방치 (useOrder.ts)

**문제**: PG 결제는 성공했지만 서버 검증이 실패한 경우, `onError`에서 alert만 표시하고 사용자가 checkout 페이지에 남아있었습니다.

**수정**: `onError`에서도 주문 상세 페이지로 이동. 주문 상세에서 현재 상태를 확인할 수 있도록.

```typescript
onError: (error, { paymentId }) => {
  handleOrderError(error);
  router.push(`/my/orders/${paymentId}`);  // 실패해도 이동
},
```

#### 3. mutateAsync 미처리 예외 (orders/[orderNumber]/page.tsx)

**문제**: `handleCancelOrder`, `handleRefund`, `handleConfirm`에서 `await mutateAsync()`를 호출하면서 try-catch가 없었습니다.

**원인**: `mutateAsync()`는 `mutate()`와 다르게 에러를 throw합니다. `onError` 콜백이 있어도 호출자에게 에러가 전파됩니다.

**수정**: 각 핸들러에 try-catch 추가.

```typescript
const handleCancelOrder = async () => {
  try {
    await cancelOrder.mutateAsync();
    setCancelModal(false);
  } catch {
    // onError 핸들러에서 alert 처리됨
  }
};
```

#### 4. 비로그인 상태에서 401 에러 (orders/page.tsx, orders/[orderNumber]/page.tsx)

**문제**: `useQuery`가 마운트 즉시 실행되어, 로그인하지 않은 상태에서도 API 호출이 발생하고 401 에러가 콘솔에 찍혔습니다.

**수정**: `enabled: isLoggedIn` 조건 추가.

```typescript
const { data, isLoading } = useQuery({
  ...queryOpts,
  enabled: isLoggedIn,  // 로그인 상태에서만 쿼리 실행
});
```

#### 5. 불필요한 router.refresh() (useOrder.ts - useConfirmOrder)

**문제**: `useConfirmOrder`의 `onSuccess`에서 `router.refresh()`를 호출했습니다.

**원인**: `router.refresh()`는 Server Component를 다시 fetch하는 Next.js 기능입니다. 하지만 우리 주문 상세 페이지는 `'use client'` Client Component이고, 데이터는 React Query로 관리합니다. `invalidateQueries`가 이미 캐시를 무효화하므로 `router.refresh()`는 아무 효과가 없습니다.

**수정**: `router.refresh()` 제거.

#### 6. 품절 상품 결제 방지 (checkout/page.tsx)

**문제**: 장바구니에 품절 상품이 포함된 채 결제가 진행될 수 있었습니다. (백엔드에서도 검증하지만 프론트에서 먼저 차단하는 것이 UX적으로 좋음)

**수정**: `hasOutOfStock` 플래그로 버튼 비활성화 + 클릭 시 alert.

```typescript
const hasOutOfStock = items.some(
  (item) => item.product.status === 'sold_out' || item.product.stockQuantity === 0
);
```

### 2차 엣지케이스 (3건) — V2 마이그레이션 후 추가 발견

#### 1. PortOne.requestPayment() 예외 미처리 (checkout/page.tsx)

**문제**: `await PortOne.requestPayment()`에 try-catch가 없어서, SDK 내부 오류 시 `isProcessingRef`가 영구 잠금되었습니다.

**원인**: `PortOne.requestPayment()`는 정상 실패(사용자 취소, 결제 실패)는 `response.code`로 반환하지만, SDK 자체 오류(네트워크 단절, 초기화 실패)는 Promise를 reject합니다.

**수정**: try-catch로 감싸서 ref 리셋 + 주문 상세로 이동.

```typescript
let response: Awaited<ReturnType<typeof PortOne.requestPayment>>;
try {
  response = await PortOne.requestPayment({ ... });
} catch {
  isProcessingRef.current = false;
  alert('결제창을 여는 중 오류가 발생했습니다.');
  router.push(`/my/orders/${order.orderNumber}`);
  return;
}
```

#### 2. SDK Address 타입 불일치 (checkout/page.tsx)

**문제**: `address: { fullAddress: form.shippingAddress }` — SDK의 `Address` 타입에 `fullAddress` 필드가 없어 TypeScript 에러.

**원인**: 포트원 공식 문서의 예시와 실제 npm 패키지의 TypeScript 타입(`@portone/browser-sdk/dist/v2/entity/Address.d.ts`)이 달랐습니다.

**수정**: SDK 타입에 맞게 `addressLine1` + `addressLine2`로 변경.

```typescript
address: { addressLine1: form.shippingAddress, addressLine2: '' },
```

#### 3. SDK 응답의 transactionId → txId (checkout/page.tsx)

**문제**: `response.transactionId`로 접근했지만, 실제 SDK의 `PaymentResponse` 타입에는 `txId` 필드만 존재.

**수정**: `response.txId`로 변경.

```typescript
verifyPayment.mutate({
  transactionId: response.txId,
  paymentId: response.paymentId,
});
```

## 6. 배운 점

1. **포트원 V1 → V2 마이그레이션**은 프론트/백 모두 변경이 필요하지만, 핵심 비즈니스 로직(금액 검증, 멱등성, 트랜잭션)은 그대로 유지됩니다. V2가 async/await를 지원하므로 프론트엔드 코드가 더 깔끔해집니다.

2. **엔티티 프로퍼티명 변경 시** `@Column({ name: '기존컬럼명' })`을 사용하면 DB 마이그레이션 없이 TypeScript 코드만 변경할 수 있습니다.

3. **`mutate()` vs `mutateAsync()`**: `mutate()`는 에러를 내부적으로 처리하고 `onError` 콜백만 호출합니다. `mutateAsync()`는 에러를 throw하므로 반드시 try-catch가 필요합니다. 모달 닫기 등 성공 후 UI 조작이 필요하면 `mutateAsync()` + try-catch를, 단순 호출이면 `mutate()`를 사용합니다.

4. **이중 클릭 방지에서 `useRef` vs `useState`**: 순수한 잠금 플래그는 리렌더링이 불필요하므로 `useRef`가 적합합니다. `useState`로 하면 상태 변경마다 리렌더링이 발생합니다.

5. **SDK 문서 vs 실제 타입**: npm 패키지의 `*.d.ts` 파일이 공식 문서 예시와 다를 수 있습니다. TypeScript를 사용하면 이런 불일치를 컴파일 타임에 잡을 수 있으므로, 타입 에러를 무시하지 말고 SDK의 실제 타입 정의를 확인해야 합니다.
