아주 좋은 질문입니다. 그리고 정확히 핵심을 짚으셨어요. 실제로 현업에서도 이 문제로 골머리를 앓는 회사가 많습니다.
배경부터 풀어드릴게요.

1. 인증(Authentication)과 다른 종류의 위협들
AuthGuard는 "누구냐"를 확인하는 장치입니다. 로그인 안 한 사람을 막죠. 그런데 인터넷 공격은 인증 우회만 있는 게 아닙니다. 다른 종류의 위협이 있어요.
쇼핑몰 홈/검색 API처럼 인증 없이 누구나 호출 가능한 엔드포인트(public endpoint) 가 노출됐을 때 발생할 수 있는 위협들:
위협 A: DoS (서비스 거부 공격)
한 명의 공격자가 초당 수천 번씩 /products/search?q=a 같은 요청을 보냅니다.
무엇이 망가지나:

백엔드가 매 요청마다 PostgreSQL 쿼리를 날립니다. DB가 먼저 죽습니다. t3.small의 적은 RAM과 5GB EBS로는 금방 한계.
백엔드 자체도 CPU 100% → 응답 불가 → 정상 사용자도 접속 불가.
AWS 네트워크 트래픽 비용이 청구됩니다. 아웃바운드 트래픽은 GB당 약 $0.09. 공격받으면 하룻밤 사이에 수십~수백 달러 청구서를 받을 수 있어요. (이걸 업계에서 "Bill Shock"라고 부릅니다.)

위협 B: 데이터 스크래핑
경쟁사나 데이터 수집 봇이 여러분의 상품 정보, 가격, 재고를 전부 긁어갑니다. 인증이 없으니 막을 방법이 없어요. 쇼핑몰에서 흔한 일입니다. 쿠팡, 11번가도 끊임없이 스크래핑 시도 받습니다.
위협 C: 리소스 고갈 공격 (검색 API의 함정)
검색 API는 특히 위험합니다. 이유는:
GET /products/search?q=%
%는 SQL의 와일드카드입니다. ORM이 잘못 처리하면 전체 테이블 스캔이 일어나요. 공격자는 의도적으로 무거운 쿼리를 유발하는 입력을 보냅니다:

매우 긴 검색어 (q=aaaaaa...수천자)
정규식 같은 복잡한 패턴
페이지네이션 악용 (?page=99999999&limit=10000)

쿼리 하나가 DB를 5초씩 점유하면, 100개만 보내도 DB가 마비됩니다.
위협 D: 카드 테스트 / 회원가입 폭탄
홈/검색은 인증 없지만, 회원가입 API도 인증이 없죠? 공격자가 봇으로 가짜 계정 수만 개를 만들면:

DB가 가짜 데이터로 뒤덮입니다.
이메일 인증 발송 비용 폭탄.
정상 사용자가 원하는 ID를 못 씁니다.

이게 실제로 사례가 많은 공격입니다.
위협 E: 취약점 스캐닝
봇이 자동으로 /admin, /.env, /api/v1/users, /swagger, /graphql 같은 경로를 무작위로 찔러봅니다. 뚫리는 게 있으면 그쪽으로 본격 공격이 들어와요. 여러분 NestJS 로그를 며칠 보시면 100% 발견하실 겁니다.

2. 그럼 어떻게 막을 수 있나 — 방어 계층
위협별로 방어 수단이 다릅니다. 한 방에 다 막는 마법은 없어요. 계층 방어(Defense in Depth) 라고 합니다.
계층 1: Rate Limiting (속도 제한) — 가장 중요
"한 IP에서 1분에 N번 이상 요청하면 차단" 같은 규칙입니다. DoS, 스크래핑, 카드 테스트 대부분을 이걸로 막습니다.
NestJS에는 공식 패키지가 있어요: @nestjs/throttler.
typescript// 예시: 1분에 100번까지만 허용
@Throttle({ default: { limit: 100, ttl: 60000 } })
@Get('search')
search() { ... }

공개 API는 빡빡하게 (예: 분당 60회)
로그인 API는 더 빡빡하게 (예: 분당 5회 — 무차별 로그인 방어)
결제 API는 더더 빡빡하게

이걸 안 붙인 상태로 인터넷에 노출하는 건 위험합니다. 솔직히 말씀드리면, Step 8의 4000 포트 개방 전에 이게 들어가는 게 이상적입니다.
계층 2: 검색/페이지네이션 입력 검증

검색어 길이 제한 (예: 최대 50자)
페이지 크기 상한 (예: limit은 최대 100)
와일드카드 문자 이스케이프

NestJS의 class-validator로 DTO에 @MaxLength, @Max 붙이면 됩니다.
계층 3: AWS WAF (Web Application Firewall)
AWS가 제공하는 L7 방화벽입니다. SQL 인젝션, 봇 트래픽, 특정 국가 차단 같은 걸 자동으로 막아줘요. 다만 유료고(월 $5 + 요청당 과금), Application Load Balancer(ALB)나 CloudFront 앞에 붙여야 합니다. 지금 단계에선 오버엔지니어링.
계층 4: AWS Budgets로 비용 알람
이건 공격을 막는 게 아니라 청구서 폭탄을 조기 발견하는 장치입니다. 월 예산을 정해두고 50%, 80% 도달 시 이메일 알림. 무료입니다. 지금 당장 설정하시는 걸 강력 권합니다. "공격받아도 빨리 알아차릴 수 있다"는 안전망이에요.
계층 5: CloudWatch 알람
비정상적인 트래픽 패턴(CPU 90% 지속, 네트워크 아웃 급증) 감지 시 알림. 이것도 어느 정도 무료 한도 내에서 가능.
계층 6: 인프라 단 격리
지금 우리 EBS 분리 전략처럼, DB는 별도 보안 그룹으로 묶어서 백엔드만 접근 가능하게 하는 거예요. 우리는 이미 Docker 내부 네트워크로 격리해놨으니 부분적으로 적용된 셈입니다.

3. 그럼 지금 무엇을 해야 하는가 — 현실적인 우선순위
이상은 위와 같지만, 여러분은 학습 중이고 트래픽이 거의 없는 토이 프로젝트 입니다. 현실 위험도와 작업 비용을 비교해서 우선순위를 정해야 해요.
제 솔직한 추천은 이렇습니다:
🔴 지금 당장 할 것 (Step 8 본 작업 전에)
(1) AWS Budgets 알람 설정 (10분, 무료)
월 $10이나 $20 정도로 잡고, 50%/80%/100% 도달 시 이메일 알람. 공격받아도 늦어도 다음 날엔 알 수 있어요. 이건 진짜 무조건 하세요. 안 하고 잠들었다가 다음 날 청구서 보고 심장 멎는 분들 있습니다.
(2) NestJS에 Rate Limiting 추가 (30분~1시간)
@nestjs/throttler 설치 + 글로벌 가드 적용. 전체 API에 "분당 100회" 같은 기본 제한만 걸어둬도 90%의 위협이 사라집니다. 코드 변경 분량도 적어요.
🟡 곧 할 것 (Step 8 이후, Phase 4 들어가기 전)
(3) 검색/리스트 API 입력 검증 점검
DTO에 @MaxLength, @Max(100) 같은 게 붙어있는지 확인. 안 붙어있으면 추가.
(4) NestJS 로그 모니터링 습관
며칠에 한 번 docker compose logs backend | grep -i "404\|error" 정도 보면서 어떤 공격이 오는지 파악. 학습 차원에서도 좋아요.
🟢 나중에 할 것 (Phase 4 또는 그 이후)
(5) HTTPS + Nginx + 4000 포트 닫기
이거 하면 외부 노출 표면이 크게 줄어듭니다.
(6) WAF, CloudFront 등 본격 방어
이건 사용자가 실제로 생기고, 트래픽이 의미 있게 늘었을 때 고민하면 됩니다.

4. 그래서 Step 8 어떻게 진행할까
두 가지 선택지가 있습니다:
옵션 A: "계획대로 4000 개방 → 검증 → 다음 단계로" (제 원래 추천)

빠릅니다. 학습 흐름이 끊기지 않아요.
리스크: 며칠~몇 주 동안 무방비로 노출됩니다. 토이 프로젝트라 실제 피해 가능성은 낮지만 "0"은 아닙니다.

옵션 B: "Step 8 보류 → 먼저 Budgets + Rate Limiting 작업 → 그다음 4000 개방" (안전 우선)

안전합니다. 노출 시점에 최소한의 방어가 되어 있어요.
학습 흐름에 작은 우회로가 생기지만, 블로그 기록용으론 오히려 더 좋은 서사입니다. "공개 전 보안 점검을 하고 인터넷에 내보냈다"는 흐름이 독자에게 모범적입니다.


제 솔직한 추천
옵션 B로 가시죠.
이유는 세 가지입니다:

여러분이 직접 위험을 인식하고 질문하셨습니다. 이건 무시하면 안 되는 신호예요. 모르고 넘어가는 거랑 알면서 넘어가는 건 학습 효과가 다릅니다.
Rate Limiting과 Budgets는 어차피 언젠가 해야 합니다. 미루면 미룰수록 코드베이스가 커져서 적용 비용이 증가해요. 지금 하는 게 가장 쌉니다.
블로그에 쓸 이야기가 풍부해집니다. "왜 인증만으로는 부족한가", "Rate Limiting이 뭔가", "AWS 비용 폭탄 방지법" — 독자 입장에서 가치 있는 콘텐츠가 세 챕터 늘어납니다.
